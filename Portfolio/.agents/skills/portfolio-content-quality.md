# portfolio-content-quality

## Purpose
Ensure portfolio copy and structure present a credible senior-quality profile, not empty template text.

## When to use
- Editing `src/app/i18n/translations.ts`.
- Creating or rewriting hero/about/projects/contact content.
- Adding new project cards, case study blocks, metrics, or process narratives.
- Converting weak placeholder copy into believable premium content.

## Core rules
- Keep content specific and outcome-oriented:
  - Problem context.
  - Approach.
  - Stack.
  - Result or measurable impact.
- For project entries, include at least:
  - What was built.
  - Why it matters.
  - Tech stack.
  - Current status or outcome.
- Use realistic, defensible metrics when data is unknown:
  - Prefer ranges or directional impact over fake exact numbers.
- Maintain consistent voice:
  - Professional.
  - Confident.
  - Clear.
- Keep EN/PL parity:
  - If a key text block changes in EN, update PL with equivalent intent.
- Fill weak sections with premium demo content when real data is unavailable, but mark internally as demo-ready text.

## Checklist
- Verify each page has clear narrative purpose.
- Verify no empty or generic one-line sections remain.
- Verify project cards are differentiated and believable.
- Verify language quality in both locales.
- Verify CTA labels are explicit and action-oriented.

## Anti-patterns
- “Lorem ipsum”-style filler or vague buzzwords.
- Same sentence pattern across all project cards.
- Unrealistic claims with no supporting context.
- Mismatched EN/PL messaging after edits.
- Dry, CV-like text blocks with no product perspective.

## Expected output
- Strong, realistic portfolio messaging across all key sections.
- Improved project credibility and narrative depth.
- Consistent bilingual content quality.

## Definition of done
- Content reads like a real professional product profile.
- Every section carries useful information and supports conversion.
- No obviously empty or low-trust copy remains.
